# NightOS Exceptional Roadmap

## What NightOS already has

- a private product posture
- a premium internal shell
- role-aware protected routes
- sync-aware reservation structures
- honest empty states
- a strong management-facing settings center

## What still separates strong from exceptional

### 1. Trusted reservation ingestion

NightOS becomes exceptional when the writer-to-reader relationship is not just possible, but dependable.

That means:

- stable booking identity
- lifecycle statuses
- reliable updated timestamps
- tested cancellation behavior
- approved production sync ownership

### 2. Direction-grade alerts

The product should surface what matters before a manager has to search for it:

- weak fill pressure
- high-value tables at risk
- missing RP attribution
- broken source freshness
- CRM relaunch opportunities

### 3. Real VIP CRM

The CRM becomes truly valuable only with:

- real client identity
- recurrence detection
- preference capture
- RP relationship history
- follow-up workflow and relaunch notes

### 4. Durable audit posture

An exceptional private platform needs:

- sign-in logs
- permission change logs
- sync-run logs
- reservation lineage
- internal activity retention policy

### 5. Direction and manager speed

The best version of NightOS should answer these questions in seconds:

- How full is tonight likely to be?
- Which RP is creating value and which one needs follow-up?
- Which clients should be protected or relaunched?
- Can we trust the writer data this week?
- What operational issue needs action before service starts?

## Recommended next priorities

1. Connect a real Supabase project and apply migrations.
2. Provision authenticated users and role rows.
3. Validate and document the writer contract with the reservation site.
4. Turn on production approval only after a successful end-to-end sync test.
5. Persist sync-run and permission events into `audit_logs`.
6. Expand CRM only after identity fields are real and consistent.
7. Add higher-level operational alerts and comparative week-over-week reporting.

## The benchmark version

NightOS becomes a club benchmark when it is:

- trusted by direction
- indispensable for managers
- fair and readable for RP performance
- disciplined enough for private operational use
- honest enough that every visible number can be defended
